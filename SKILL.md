name: teaching
description: 教学时使用。不一定是用户直接要求教学时才算，如果用户要求你讲讲某一个主题（比较大，需要多次追问），这也算教学。
---

这个skill的目标就是提高用户学习的体验，让用户在阅读教学内容的时候更方便，我们讲的是一种实现这个目标的参考方法，如果你还有其他想法能实现这个目标，你可以自由发挥。

想象一下一个人类教师在教授概念时是怎样的：教师会在黑板上写东西，当需要补充时，教师就会在对应的位置补上，而不是在黑板的末尾追加写入。

当你教授概念时，也是类似的，你要创建一个文件，比如md文档，在里面写入内容来教授用户，这样在需要对之前的内容进行修正时，只需要修改对应位置

展开来说，就是：
- 当用户要求补充某个概念时，优先在对应位置新增或拆分小节，而不是在文末追加新章节。
- 当用户指出结构问题时，允许并优先对原有结构进行重组，而不是追加内容。
- 新增内容时默认一个核心概念一个小节结构，避免多个概念堆在同一节。
- 用户的新问题、追问、补充说明，应优先补到对应概念所在的位置；默认不要按提问时间顺序在文末继续追加。
- 如果一个回答里同时带出了额外问题，就在该回答后面直接继续补解释，不要把这个额外问题拆到文档后面另起一节。

---

知识是有结构的，你要用到md的标题语法来构建这个结构。

标题通常不建议写用户的问题，而是写知识。问题与回答要收集进对应“问答”section。

假设一个主题的知识结构如下：
```
topic S
|
|--subtopic A
| |
| |--subtopic A1
| | |
| | |--subtopic A1.1
| | |
| | |--subtopic A1.2
| |
| |--subtopic A2
|
|--subtopic B
|
|--subtopic C

```

那么对应的md文档结构应当如下：
```
# topic S
## subtopic A
### subtopic A1.1
### subtopic A1.2
## subtopic B
## subtopic C
```

对于假设用户问了 subtopic A 和 subtopic B 的区别，subtopic A1.1 和 subtopic A1.2 的区别，那么就添加 Q&A 章节：
```
# topic S
## subtopic A
### subtopic A1.1
### subtopic A1.2
### Q&A about A1.1 & A1.2
## subtopic B
## Q&A about A&B
## subtopic C
```

假设 subtopic B 又引出了关于另一个比较大的主题 K 的讨论（K并不隶属于S），那么关于K的讨论要放到另外一个文档，并在当前文档中引用

---

用户通常是在编辑器里面查看md文档，有折叠某个md块的功能，所以你不需要担心内容太长影响阅读的问题，用户不想读的时候可以折叠的。

写完文档后，回复应当简短，不要再重复文档中的内容，也就是说，所有的教学的内容都应当写在文档中，而不是一部分在文档，一部分又在你后续的回复中。文档中已经传达过的信息，你不要再输出一遍。

提供一个最简单版本的例子会有助于理解。

类比是很好的教学工具。
