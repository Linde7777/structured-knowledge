# 概述
这个 skill 的目标是提高长期学习和讨论的体验。用语言去描述出你所有应该做的事情是很难的，下面要描述的方法是一种很好参考，并没有涵盖所有情况，如果你感觉“有些事情做了是有利于实现这个目标的”，但是本文档没有描述到，甚至可能违反了本文档描述的内容，那么你可以去做。重点是实现这个目标。

当用户围绕一个主题反复追问、比较、纠错、扩展时，你要维护一份结构化的 Markdown 文档，而不是按时间顺序追加聊天记录；不要因为一个问题在时间线上出现得晚，就把它追加到文档末尾，它在结构上属于哪里，就放到哪里。

- 当用户要求补充某个概念时，优先在对应位置新增或拆分小节，而不是在文末追加新章节。
- 新增内容时默认“一个核心概念一个小节”，避免多个概念堆在同一节。
- 当用户指出结构问题时，要对原有结构进行重组，而不是追加内容。

## 细则

要用md标题语法来表达这个结构。

标题通常不建议写用户的问题，而是写知识对象或决策点。

如果能通过补充知识对象就能回答用户的问题，就没有必要再去专门写一个“问题与回答” section，如果要写，参考后面提到的`问答section要求`

假设一个主题的知识结构如下：
```
topic S
├── subtopic A
│   ├── subtopic A1
│   │   ├── subtopic A1.1
│   │   └── subtopic A1.2
│   └── subtopic A2
├── subtopic B
└── subtopic C
```

那么对应的md文档的名字得是跟topic S相关的，然后其内容结构应当如下：
```
# subtopic A
## subtopic A1
### subtopic A1.1
### subtopic A1.2
## subtopic A2
# subtopic B
# subtopic C
```

如果在一节内容里，你还想表达一些知识的层级，但是又感觉用md标题会太重，你可以用缩进来表达这种关系。不过缩进一个纯文本通常会显得有点怪，所以一般推荐用 bullet point。示例如下：
```
- content 1
  - subcontent 1.1
    - subcontent 1.1.1
  - subcontent 1.2
```

如果你想在一节内容的内部做个划分，但是感觉用md标题又太重，你可以用`---`来划分。

---

如果讨论引出了另一个较大的主题，且它不隶属于当前主题，应创建或维护另一个文档，并在当前文档中引用（哪里引出这个主题的，就在哪里引用）。

如果用户在阅读你写的文档中提出了一个问题，然后你通过补充文档回答了这个问题，记得要在引起用户疑问的地方，去引用你刚补充的内容（简短引用，比如引用其标题）。类似的，如果用户提出疑问的地方，文档中其实原本就有地方是回答了的，那么你要补上对应的引用。

### 问答section要求

问答 section 要放在它所比较或讨论的对象的最近共同父级下，并且和这些对象保持同级。

如果用户问的是 `subtopic A1.1` 和 `subtopic A1.2` 的区别，那么 `Q&A about A1.1 & A1.2` 应该放在它们的共同父级 `subtopic A1` 下

如果用户问的是 `subtopic A` 和 `subtopic B` 的区别，那么 `Q&A about A & B` 应该放在 `A` 和 `B` 的共同父级 `topic S`（整个文档的主题） 下

对应结构示例：
```
# subtopic A
## subtopic A1
### subtopic A1.1
### subtopic A1.2
### Q&A about A1.1 & A1.2
## subtopic A2
# subtopic B
# Q&A about A & B
# subtopic C
```

# 其他注意事项

用户通常是在编辑器里面查看md文档，有折叠某个md块的功能，所以你不需要担心内容太长影响阅读的问题，用户不想读的时候可以折叠的。

为了更加方便地显示出你做了哪些改动，在你改动后先不要commit，这样用户在编辑器里能看出你改了哪里，正要改下一版的时候，先commit，然后再开始改。

写完或更新文档后，回复要简短，文档中已经传达过的信息，不要再输出一遍，用户自己会去看文档的

<teaching>
提供一个最简单版本的例子会有助于用户理解。

类比是很好的教学工具。

如果用户反馈说难以理解，那么通常推荐讲解一下“这个知识的再往下一层的知识”，比如用户难以理解关于前端的某一个知识，那么就要讲一下相关的浏览器的知识。
</teaching>
