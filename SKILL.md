# 概述

**目标：让用户在长期学习、讨论和反复追问同一主题时，获得更好的知识组织体验（便于理解、扩展）。**

**本文档描述的规则是启发式的，并没有穷尽所有应当做或者不做的事。在实际执行时，应优先去想要怎样做才有利于实现这个目标。如果某个做法虽然没有被明确写入本文档，但明显有助于实现这个目标，可以主动采用；如果某条具体规则在当前场景下反而妨碍了这个目标，也可以调整。重点是实现目标**

当用户围绕一个主题反复追问、比较、纠错、扩展时，你要维护一份结构化的 Markdown 文档，而不是按时间顺序追加聊天记录；不要因为一个问题在时间线上出现得晚，就把它追加到文档末尾，它在结构上属于哪里，就放到哪里。

- 当用户要求补充某个概念时，优先在对应位置新增或拆分小节，而不是在文末追加新章节。
- 新增内容时默认“一个核心概念一个小节”，避免多个概念堆在同一节。
- 当用户指出结构问题时，要对原有结构进行重组，而不是追加内容。

## 细则

要用md标题语法来表达这个结构。

标题通常不建议写用户的提问，而是写知识对象或决策点。

如果能通过补充知识对象就能回答用户的问题，就没有必要再去专门写一个“问题与回答” section，如果要写，参考后面提到的`问答section要求`

假设一个主题的知识结构如下：
```
topic S
├── subtopic A
│   ├── subtopic A1
│   │   ├── subtopic A1.1
│   │   └── subtopic A1.2
│   └── subtopic A2
├── subtopic B
└── subtopic C
```

那么对应的md文档的名字得是跟topic S相关的，然后其内容结构应当如下：
```
# subtopic A
## subtopic A1
### subtopic A1.1
### subtopic A1.2
## subtopic A2
# subtopic B
# subtopic C
```

如果在一节内容里，你还想表达一些知识的层级，但是又感觉用md标题会太重，你可以用缩进来表达这种关系。不过缩进一个纯文本通常会显得有点怪，所以一般推荐用 bullet point。示例如下：
```
- content 1
  - subcontent 1.1
    - subcontent 1.1.1
  - subcontent 1.2
```

如果你想在一节内容的内部做个划分，但是感觉用md标题又太重，你可以用`---`来划分。

---

用 Minto 金字塔方法去写：先给结论或判断，再展开原因、结构和细节。

写解释性文档时，优先回答“为什么要这样设计 / 为什么需要这个东西”，再介绍“它由哪些部分组成 / 每个部分做什么”。用户通常不是卡在名词定义上，而是卡在“为什么这里需要它、如果没有它会怎样、它在解决什么问题”。

比如讲火箭结构时，不要一上来平铺“燃料箱、发动机、整流罩、控制系统分别是什么”。更好的顺序是：

```
火箭要把载荷送入轨道
├── 必须产生足够推力，所以需要发动机
├── 必须持续供给推进剂，所以需要燃料箱
├── 必须保护载荷穿过大气层，所以需要整流罩
└── 必须保持姿态和轨迹，所以需要控制系统
```

也就是说，组件说明应当服务于“为什么需要这些组件”这个主线，而不是把组件功能孤立罗列出来。

如果文档一开始就按“为什么 → 怎么做 → 细节”的顺序展开，很多读者疑问会自然被前文解掉，不需要额外补交叉引用。

交叉引用是补救机制：如果用户在阅读你写的文档时对某一句、某一段提出问题，而答案被补充在另一个更合适的结构位置，就要回到引起疑问的地方补一个简短引用（比如引用其标题）。如果用户疑惑的答案其实原本已经在文档别处存在，也要在疑惑处补上对应引用。

补引用时还要检查原句本身是否容易误导。如果原句的表达顺序让读者先看到“做什么”，但暂时不知道“为什么要这么做”，优先改写原句，让它先交代动机，这样就讲清楚了，可能都不需要去补引用了。

---

如果讨论引出了另一个较大的主题，且它不隶属于当前主题，应创建或维护另一个文档，并在当前文档中引用（哪里引出这个主题的，就在哪里引用）。

### 问答section要求

问答 section 要放在它所比较或讨论的对象的最近共同父级下，并且和这些对象保持同级。

如果用户问的是 `subtopic A1.1` 和 `subtopic A1.2` 的区别，那么 `Q&A about A1.1 & A1.2` 应该放在它们的共同父级 `subtopic A1` 下

如果用户问的是 `subtopic A` 和 `subtopic B` 的区别，那么 `Q&A about A & B` 应该放在 `A` 和 `B` 的共同父级 `topic S`（整个文档的主题） 下

对应结构示例：
```
# subtopic A
## subtopic A1
### subtopic A1.1
### subtopic A1.2
### Q&A about A1.1 & A1.2
## subtopic A2
# subtopic B
# Q&A about A & B
# subtopic C
```

# 其他注意事项

用户通常是在编辑器里面查看md文档，有折叠某个md块的功能，所以你不需要担心内容太长影响阅读的问题，用户不想读的时候可以折叠的。

为了更加方便地显示出你做了哪些改动，在你改动后先不要commit，这样用户能在编辑器里看出你改了哪里，正要改下一版的时候，先commit，然后再开始改。

写完或更新文档后，回复要简短，文档中已经传达过的信息，不要再输出一遍，用户自己会去看文档的

<teaching>
提供一个最简单版本的例子会有助于用户理解。

类比是很好的教学工具。

如果用户反馈说难以理解，那么通常推荐讲解一下“这个知识的再往下一层的知识”，比如用户难以理解关于前端的某一个知识，那么就要讲一下相关的浏览器的知识。
</teaching>
